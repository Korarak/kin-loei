Mark 1 → Mark 2

เปลี่ยนแกน AI มาเป็น Gemini

Mark 1 ใช้หลายเครื่องมือต่อกัน (OCR + Vision + NLP + Rule Engine แยกชิ้น) ส่วน Mark 2 ยุบรวมงานความเข้าใจภาพและภาษาทั้งหมดมาที่ Gemini 3 Flash ตัวเดียว ทำให้ระบบเรียบง่ายขึ้น พัฒนาเร็วขึ้น และฉลาดขึ้น

✨ ทำไม Gemini 3 Flash เหมาะกับกินเลย

→Multimodal ในตัว — รับภาพฉลาก/อาหารเข้าไปตรงๆ ได้เลย ไม่ต้องต่อ OCR แยก
→Search Grounding — สืบค้นข้อมูลส่วนผสม/สารก่อภูมิแพ้สดใหม่จาก Google ได้
→Structured Output — ให้ตอบเป็น JSON ที่แอปนำไปแสดงผล 🟢🟡🔴 ได้ทันที
→Free Tier — ทดลองและทำ MVP ได้ฟรี เหมาะกับทีมนักศึกษา
→Context Caching — แคชโปรไฟล์/ฐานข้อมูลโรค ลดต้นทุนเรียกซ้ำ
→เข้าใจภาษาไทย — อ่านฉลากไทยและตอบภาษาไทยได้ดี รองรับอีสานได้

Architecture Decision

ใช้ Gemini ทั้งหมด หรือผสม ML เราเอง?

มีสองทางเลือกหลักในการวางสถาปัตยกรรม AI แต่ละทางมีข้อดีต่างกัน — ขึ้นกับเป้าหมายและทรัพยากรของทีม

แนะนำสำหรับเริ่มต้น
Approach A

Gemini-only

ให้ Gemini 3 Flash จัดการทุกอย่าง — อ่านภาพ ระบุส่วนผสม สืบค้น ประเมินผล

  • พัฒนาเร็วที่สุด ไม่ต้องเทรนหรือดูแลโมเดลเอง
  • ความแม่นยำสูงทันทีตั้งแต่วันแรก เพราะเป็นโมเดลระดับ frontier
  • อัปเกรดความสามารถตามที่ Google พัฒนาโมเดลใหม่
  • เหมาะกับทีมเล็กที่อยากออก MVP ขึ้น Play Store เร็ว
เหมาะถ้า: อยากเห็นผลงานใช้งานจริงเร็ว เน้นคุณภาพการวิเคราะห์ และยอมรับการพึ่งพา API ภายนอก
Approach B

Hybrid — Gemini + ML เราเอง

ใช้ ML เบาบนเครื่องคัดกรองก่อน แล้วส่งต่อ Gemini เฉพาะเคสที่ต้องวิเคราะห์ลึก

  • On-device ML (เช่น บาร์โค้ด/หมวดอาหาร) ทำงานฟรีและออฟไลน์
  • เรียก Gemini เฉพาะเมื่อจำเป็น ลดต้นทุน API ลงมาก
  • เป็นจุดขายเชิงวิชาการ — ทีมได้แสดงทักษะสร้างโมเดลเอง
  • ควบคุมข้อมูลและความเป็นส่วนตัวได้มากกว่าในบางขั้นตอน
เหมาะถ้า: อยากโชว์ทักษะ ML ของทีม ลดต้นทุนระยะยาว และมีเวลาพัฒนา/เทรนโมเดลเพิ่ม

Feasibility Study

เป็นไปได้แค่ไหนที่จะขึ้น Play Store

ประเมินความเป็นไปได้จริงของการทำ Hybrid App แล้วเผยแพร่บน Google Play — ทั้งด้านเทคนิคและด้านนโยบาย

✅ ด้านเทคนิค ทำได้จริง

  • Flutter build เป็นไฟล์ .aab/.apk ขึ้น Play Store ได้โดยตรง
  • Gemini API เรียกผ่าน HTTPS ได้จากแอปทุกแพลตฟอร์ม
  • กล้อง + ออฟไลน์แคช เป็นความสามารถมาตรฐานของ Flutter
  • MVP ใช้งานได้จริงทำได้ภายในกรอบเวลาโครงการ

⚠️ ด้านนโยบาย/ความปลอดภัย ต้องวางแผน

  • อย่าฝัง API Key ในแอป — ต้องเรียกผ่าน Backend proxy เสมอ
  • ข้อมูลสุขภาพเป็นข้อมูลอ่อนไหว ต้องมี Privacy Policy + PDPA
  • Play Store ต้องการคำประกาศ Data Safety ที่ชัดเจน
  • ระบุชัดว่าเป็นผู้ช่วยให้ข้อมูล ไม่ใช่อุปกรณ์การแพทย์
⚠️

หลักการสำคัญด้านการแพทย์

กินเลยทำหน้าที่เป็นผู้ช่วยให้ข้อมูล ไม่ใช่เครื่องมือวินิจฉัยทางการแพทย์ ทุกคำแนะนำอยู่บนพื้นฐาน "เตือนและแนะนำ" พร้อมระบุชัดเจนว่าผู้ใช้ควรปรึกษาแพทย์หรือเภสัชกรสำหรับการตัดสินใจที่สำคัญ — ข้อความนี้ควรปรากฏทั้งในแอปและในหน้า Play Store

Guiding Framework

Smart · Green · Safe · Creative

กรอบสี่ด้านที่ Gemini ช่วยให้ทำได้จริงและดียิ่งขึ้นในเวอร์ชัน Mark 2

Smart

ฉลาดขึ้นด้วย Gemini

โมเดล frontier วิเคราะห์ฉลากที่ซับซ้อนและสืบค้นข้อมูลส่วนผสมที่ไม่รู้จักได้เอง

  • เข้าใจฉลากที่เขียนกำกวมหรือถ่ายภาพไม่ชัดได้ดีกว่า OCR ธรรมดา
  • สืบค้นสารก่อภูมิแพ้ที่เพิ่งมีข้อมูลใหม่ผ่าน Search Grounding
  • แนะนำทางเลือกที่เหมาะกว่าตามบริบทของผู้ใช้
Green

สุขภาพคู่ความยั่งยืน

ให้ Gemini ประเมินและแนะนำในมิติสิ่งแวดล้อมควบคู่กับสุขภาพ

  • แนะนำสินค้าท้องถิ่นจังหวัดเลยที่สดใหม่และเดินทางสั้น
  • ลดของเสียจากการซื้อผิด ส่งเสริมเกษตรอินทรีย์ในพื้นที่
  • ให้ข้อมูลโดยประมาณด้านผลกระทบสิ่งแวดล้อมของผลิตภัณฑ์
Safe

ปลอดภัยและมีจริยธรรม

จุดที่ละเอียดอ่อนที่สุดคือข้อมูลสุขภาพ ระบบต้องวางใจได้และโปร่งใส

  • แนะนำและเตือนเท่านั้น — ไม่วินิจฉัย แนะให้ปรึกษาแพทย์เสมอ
  • เรียก Gemini ผ่าน Backend proxy ไม่เปิดเผย API Key
  • ทำตาม PDPA + Play Data Safety ผู้ใช้ลบข้อมูลตัวเองได้
Creative

จากบุคคลสู่ระบบนิเวศ

ความสามารถของ Gemini เปิดทางให้ฟีเจอร์ที่ทำได้ยากมาก่อน

  • โหมดสนทนาด้วยเสียงสำหรับผู้สูงอายุ (Gemini Live ในอนาคต)
  • รองรับภาษาอีสาน เพราะ Gemini เข้าใจหลายภาษา
  • เปิด API ให้ รพ.สต./โรงเรียนในจังหวัดเลยต่อยอด
